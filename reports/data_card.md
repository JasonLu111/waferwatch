# WaferWatch Data Card

## 1. Dataset Overview

This data card documents the demo datasets used in **WaferWatch: Cost-Sensitive Fab Anomaly Monitoring and Evidence-Grounded Root-Cause Triage**.

The current WaferWatch repository uses synthetic demo data generated inside the project. The data is designed to test the end-to-end pipeline architecture, including:

- Data validation
- Data ingestion
- Data cleaning
- Feature engineering
- SPC feature construction
- Feature selection
- Model training
- Model evaluation
- Cost-sensitive thresholding
- Batch prediction
- Drift monitoring
- Performance monitoring
- Alert summary generation

The current data should be interpreted as controlled demo data, not real production data.

---

## 2. Dataset Purpose

The demo data is used to verify that WaferWatch can run as a complete manufacturing anomaly monitoring workflow.

The data is designed to support three main goals:

1. **Pipeline validation**  
   Confirm that each project module can read, transform, save, and report data correctly.

2. **Modeling demonstration**  
   Train baseline models and compare feature strategies under a controlled anomaly mechanism.

3. **Monitoring demonstration**  
   Simulate data drift and performance degradation to verify monitoring alerts.

The data is not intended to represent real fab performance, real tool behavior, or actual production yield outcomes.

## 4. Data Generation Method

The current synthetic SPC dataset is generated by:

```text
src/features/spc_features.py
```

The generator creates:

- Lot IDs
- Timestamps
- Multiple synthetic sensor columns
- A binary `pass_fail_label`
- Injected anomaly patterns in selected lots

The synthetic sensor values are generated from normal distributions. Some rows are then modified by injecting abnormal sensor shifts. These shifted rows are assigned the risky label.

This design makes the anomaly mechanism intentionally learnable by SPC-derived features.

---

## 5. Unit of Observation

The current unit of observation is:

```text
one row = one production lot in the demo setting
```

Each row contains:

- Lot identifier
- Timestamp
- Sensor readings or engineered features
- Optional label
- Optional prediction output
- Optional monitoring output

The current project does not model wafer-level, die-level, chamber-level, or step-level granularity yet.

---

## 6. Label Definition

The current label column is:

```text
pass_fail_label
```

| Label | Meaning |
|---:|---|
| 0 | Normal or non-risky lot in the demo setting |
| 1 | Risky or anomalous lot in the demo setting |

Important limitation:

The current labels are synthetic. They are generated by the project code and are not based on real fab yield, metrology, inspection, or downstream quality outcomes.

## 7. Feature Groups

WaferWatch currently uses three major feature groups.

### 7.1 Raw Sensor Columns

Synthetic raw sensor columns follow the naming pattern:

```text
sensor_001
sensor_002
sensor_003
...
```

These columns represent simplified process or equipment signals in the demo setting.

### 7.2 Sensor Aggregate Features

Sensor aggregate features summarize row-level sensor behavior:

| Feature | Meaning |
|---|---|
| `sensor_mean` | Mean across sensor columns |
| `sensor_std` | Standard deviation across sensor columns |
| `sensor_min` | Minimum sensor value |
| `sensor_max` | Maximum sensor value |
| `sensor_missing_count` | Number of missing sensor values |
| `sensor_missing_ratio` | Missing sensor value ratio |

### 7.3 SPC Features

SPC features compare sensor values against reference control limits:

| Feature | Meaning |
|---|---|
| `spc_violation_count` | Number of sensor values outside control limits |
| `spc_violation_ratio` | Ratio of sensors outside control limits |
| `spc_max_abs_zscore` | Maximum absolute standardized sensor deviation |
| `spc_any_violation` | Whether any SPC violation occurred |
| `spc_top_violating_sensor` | Sensor with strongest deviation |

## 9. Missing Values

The current demo data has limited missing-value complexity.

The cleaning module supports:

- Duplicate-row removal
- High-missingness column removal
- Median imputation for numeric columns

The current synthetic SPC dataset does not fully stress-test realistic missingness patterns such as:

- Tool-specific missing values
- Recipe-dependent missing sensors
- Random sensor dropouts
- Maintenance-related missingness
- Late-arriving labels

Future work should add controlled missingness experiments.

---

## 10. Class Balance

The SPC demo dataset uses a synthetic anomaly injection rate of approximately 20%.

This creates an imbalanced but still learnable binary classification task.

In real manufacturing anomaly detection, abnormal or yield-risk events may be much rarer than 20%. Therefore, future experiments should test:

- Lower anomaly rates
- More severe class imbalance
- Temporal splits
- Product or recipe group splits
- Cost-sensitive evaluation under rare-event conditions

---

## 11. Train/Test Splitting

The current baseline training scripts use deterministic train/test splits with a fixed random seed.

This is suitable for early pipeline validation, but it is not enough for production-grade validation.

Future validation should include:

- Time-based splits.
- Tool-based splits.
- Product-based splits.
- Recipe-based splits.
- Cross-period validation.
- Evaluation under drifted data.
- Evaluation on new anomaly mechanisms.

---

## 12. Drift Data

The drift monitoring demo uses:

```text
src/monitoring/drift.py
```

It splits the selected feature table into:

- Reference period
- Current period

Then it injects synthetic current-period drift into selected features.

The generated drift table is:

```text
demo_drift_table.csv
```

Current monitored drift metrics include:

- Standardized mean shift
- Standard deviation ratio
- Missing rate shift
- Population Stability Index

The drift data is synthetic and should only be used to verify monitoring logic.

---

## 13. Performance Monitoring Data

The performance monitoring demo uses:

```text
src/monitoring/performance.py
```

It simulates current-period performance degradation by weakening the relationship between SPC signals and labels.

This is used to test whether WaferWatch can detect:

- Recall degradation
- PR-AUC degradation
- False alarm increase
- Confusion matrix changes

The performance degradation is artificial and should not be interpreted as a real production event.

---

## 14. Data Limitations

The current data has the following limitations:

- It is synthetic.
- It is small.
- It uses simplified sensor behavior.
- It does not contain real fab process context.
- It does not include real tool IDs, chambers, recipes, operators, products, or routes.
- It does not include real maintenance logs or engineering notes.
- It does not include real metrology, inspection, or yield outcomes.
- It does not include delayed labels.
- It does not include lot genealogy.
- It does not include multi-step process dependencies.
- It does not include realistic temporal autocorrelation.
- It does not include privacy, security, or access-control constraints from real manufacturing systems.

---

## 15. Data Ethics and Safety

The current project does not use real personal data or proprietary production data.

If WaferWatch is extended to real manufacturing data, the following issues must be considered:

- Data access control
- Proprietary process protection
- Audit logging
- Secure storage of production data
- Responsible use of model outputs
- Avoiding unsupported root-cause claims
- Human review before operational escalation
- Separation between decision support and automated control
- Validation before any production-facing deployment

## 18. Reproducibility

Most demo data files can be regenerated by running the pipeline modules in order.

Recommended order:

```bash
python -m src.data.ingest
python -m src.data.clean
python -m src.features.spc_features
python -m src.features.build_features
python -m src.features.feature_selection
python -m src.models.compare
python -m src.models.threshold
python -m src.models.predict
python -m src.monitoring.drift
python -m src.monitoring.performance
python -m src.monitoring.alerts
```

The project uses fixed random seeds in demo generators where appropriate to improve reproducibility.

## 20. Data Card Summary

The current WaferWatch data layer is a controlled synthetic demonstration.

It is useful for validating the architecture and showing how a manufacturing anomaly monitoring system can be structured.

It should not be presented as real production data or evidence of real fab performance.

The correct interpretation is:

> WaferWatch currently uses synthetic demo data to validate an end-to-end manufacturing AI monitoring architecture.

The incorrect interpretation is:

> WaferWatch has already proven real-world fab anomaly detection performance.
