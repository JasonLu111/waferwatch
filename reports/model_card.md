# WaferWatch Model Card

## 1. Model Overview

**Project name:** WaferWatch  
**Subtitle:** Cost-Sensitive Fab Anomaly Monitoring and Evidence-Grounded Root-Cause Triage

WaferWatch is a manufacturing AI decision-support project for semiconductor-style anomaly monitoring. The current model pipeline predicts lot-level risk scores from engineered sensor and SPC features, then supports cost-sensitive escalation decisions for engineering review.

The model is designed as a human-in-the-loop decision-support system. It is not intended to automatically stop production, classify root cause, or replace engineering judgment.

## 3. Intended Use

WaferWatch is intended to support:

- Lot-level anomaly risk scoring.
- Engineering review prioritization.
- Cost-sensitive escalation decisions.
- Monitoring of feature distribution drift.
- Monitoring of model performance after labels become available.
- Generating structured evidence for later root-cause triage workflows.

The system is designed for portfolio and research demonstration purposes at this stage. It is not production validated.

---

## 4. Not Intended Use

WaferWatch should not be used for:

- Fully automated production hold or release decisions.
- Final root-cause diagnosis without engineering review.
- Safety-critical decisions without formal validation.
- Real fab deployment without production-scale data validation.
- Model performance claims on real manufacturing data.
- Cross-tool, cross-product, or cross-recipe generalization claims without additional experiments.

## 7. Feature Engineering

WaferWatch currently includes three levels of feature engineering.

### 7.1 Sensor Aggregate Features

These features summarize row-level sensor behavior:

| Feature | Meaning |
|---|---|
| `sensor_mean` | Mean value across sensor columns |
| `sensor_std` | Standard deviation across sensor columns |
| `sensor_min` | Minimum sensor value |
| `sensor_max` | Maximum sensor value |
| `sensor_missing_count` | Number of missing sensor values |
| `sensor_missing_ratio` | Ratio of missing sensor values |

### 7.2 SPC Features

SPC means Statistical Process Control. These features compare sensor behavior against reference control limits.

| Feature | Meaning |
|---|---|
| `spc_violation_count` | Number of sensor values outside control limits |
| `spc_violation_ratio` | Ratio of sensors violating control limits |
| `spc_max_abs_zscore` | Maximum absolute standardized deviation |
| `spc_any_violation` | Whether any SPC violation occurred |
| `spc_top_violating_sensor` | Sensor with the strongest SPC deviation |

### 7.3 Selected Features

The feature selection module removes non-numeric, constant, near-constant, high-missingness, and highly correlated features.

The current selected feature set is:

- `sensor_mean`
- `sensor_std`
- `sensor_min`
- `sensor_max`
- `spc_violation_count`
- `spc_max_abs_zscore`

Removed features include:

| Removed feature | Reason |
|---|---|
| `spc_top_violating_sensor` | Non-numeric feature excluded from current baseline |
| `sensor_missing_count` | Constant feature in current demo |
| `sensor_missing_ratio` | Constant feature in current demo |
| `spc_any_violation` | Highly correlated with other SPC features |
| `spc_violation_ratio` | Highly correlated with other SPC features |

## 8. Model Type

The current baseline model family is:

```text

```

The model is trained using:

## 9. Evaluation Metrics

WaferWatch does not rely on accuracy alone. This is important because anomaly detection is usually imbalanced and operationally cost-sensitive.

The current evaluation module reports:

| Metric | Reason for inclusion |
|---|---|
| Accuracy | General correctness, but insufficient alone |
| Precision | Measures false alarm burden |
| Recall | Measures ability to catch risky lots |
| F1 | Balances precision and recall |
| Balanced accuracy | More useful under class imbalance |
| Matthews correlation coefficient | Robust binary classification summary |
| ROC-AUC | Ranking quality across thresholds |
| PR-AUC | More informative under class imbalance |
| Precision@K | Quality of top engineering review list |
| Recall@K | Whether top-K captures true risky lots |
| False alarms per 100 lots | Operational review burden |
| Confusion matrix | Direct TP / FP / FN / TN interpretation |

## 11. Interpretation of Evaluation Results

The controlled demo shows that SPC-derived features improve anomaly detection when the anomaly mechanism is reflected in sensor excursions.

However, this result must be interpreted carefully:

- The dataset is synthetic.
- The SPC anomalies are intentionally injected.
- The test set is small.
- Perfect scores are expected in a controlled demonstration when engineered SPC features directly encode the anomaly mechanism.
- These results do not prove real fab deployment performance.

The correct conclusion is:

> SPC-derived features can provide useful manufacturing-relevant signals in a controlled anomaly monitoring pipeline.

The incorrect conclusion would be:

> The current model is production-ready or guaranteed to perform well on real fab data.

## 13. Isolation Forest Threshold Tuning

Isolation Forest is useful when confirmed failure labels are rare or delayed, but its default threshold can generate too many false alarms. WaferWatch therefore evaluates top-K review policies and false-alarm budget constraints.

|---|---:|---:|---:|---:|---:|
| default_isolation_forest_threshold | 12 | 0.416667 | 1.000000 | 0.588235 | 29.166667 |
| top_3_review | 3 | 1.000000 | 0.600000 | 0.750000 | 0.000000 |
| top_5_review | 5 | 1.000000 | 1.000000 | 1.000000 | 0.000000 |

| top_10_review | 10 | 0.500000 | 1.000000 | 0.666667 | 20.833333 |

Under a budget of 10 false alarms per 100 lots, the selected policy is:


This policy preserves full recall and avoids false alarms in the controlled synthetic demo. The result should be interpreted as workflow validation, not production performance.

## 14. Cost-Sensitive Thresholding

WaferWatch includes a thresholding module because the default 0.5 threshold is often not appropriate for manufacturing anomaly detection.

The current thresholding demo compares:

| Threshold strategy | Purpose |
|---|---|
| Default threshold | Uses 0.50 as a standard classifier cutoff |
| Simple cost-sensitive threshold | Uses expected cost logic |
| Best threshold by realized cost | Searches thresholds to minimize demo operational cost |

The thresholding demo produced:

| Strategy | Threshold | Total cost |
|---|---:|---:|
| Default threshold | 0.50 | 29.0 |
| Simple cost-sensitive threshold | 0.05 | 37.0 |
| Best threshold by realized cost | 0.42 | 10.0 |

This shows that maximizing recall alone is not always optimal. A very low threshold can catch more risky lots but may create excessive false alarms and engineering review burden.

## 17. Ethical and Operational Considerations

WaferWatch should support engineers, not replace them.

Important considerations:

- False negatives can allow risky lots to pass without review.
- False positives can create alert fatigue and unnecessary engineering workload.
- Cost assumptions should be reviewed with domain experts.
- Model explanations should be paired with process context.
- Monitoring alerts should be reviewed by qualified engineers.
- Production use would require audit trails, access control, validation, and incident handling.
